Customize EDS blocks and styling

EDS renders any markdown table as a block named after its first cell — and blocks are where all your custom branding lives. This task walks through editing an existing block, adding a new one, and updating site-wide styling, including how the changes propagate through GitHub to aem.live.

Before you begin:

  • You have a working EDS repository with the boilerplate blocks present at blocks//.
  • You can install Node.js dependencies for local preview if you want one (aem up runs a proxy server).
  • You understand basic CSS and modern JavaScript (ES2022 modules).

Blocks are the building blocks of an EDS site. Each block is a folder under blocks/ with two files: a stylesheet and a JavaScript module. The framework loads them automatically when it encounters a <div class="<name>"> in the page.

The transform Action rewrites DITA notes, tables, and a handful of other elements into block divs, so the cycle from "DITA author wants a new note variant" to "branded note appears on the page" is a transform update plus a new block folder.

  1. Decide which block you want to edit.The POP-branded blocks shipped with the repository are blocks/{note,warning,tip,caution,important,table,sidenav,header,footer}/. Each is independent — editing one will not affect the others.
  2. Edit the block's CSS file at blocks//.css.The block's outer element has the class <name> plus the standard EDS decoration classes (block, data-block-name="<name>"). You can target it directly.

    For example, to make warnings have a stronger left-border accent and dark text:

    .warning {
  border-left: 4px solid var(--pop-color-warning);
  background: var(--pop-warning-tint);
  color: var(--pop-color-text);
  padding: 1rem 1.25rem;
  border-radius: 0.25rem;
}
.warning::before {
  content: "⚠";
  margin-right: 0.5rem;
  font-weight: 700;
}
  3. If the block needs runtime behavior, edit blocks//.js.The default export is invoked with the block's root element when the framework decorates the block. The function can be async.

    For instance, the table block's job is to promote the <div class="table"> structure into a real <table> with <thead> and <tbody>:

    export default function decorate(block) {
  const rows = [...block.children];
  const table = document.createElement('table');
  const thead = document.createElement('thead');
  const headerRow = document.createElement('tr');
  for (const cell of rows[0].children) {
    const th = document.createElement('th');
    th.innerHTML = cell.innerHTML;
    headerRow.append(th);
  }
  thead.append(headerRow);
  table.append(thead);
  const tbody = document.createElement('tbody');
  for (const row of rows.slice(1)) {
    const tr = document.createElement('tr');
    for (const cell of row.children) {
      const td = document.createElement('td');
      td.innerHTML = cell.innerHTML;
      tr.append(td);
    }
    tbody.append(tr);
  }
  table.append(tbody);
  block.replaceChildren(table);
}
  4. (Optional) Preview locally with the AEM CLI.The aem-cli runs a local proxy that serves your local blocks/, scripts/, and styles/ on top of the live aem.page content. Run from the repository root:
    aem up
    The proxy listens on http://localhost:3000 and hot-reloads on any local file change.
  5. Commit and push.
    git add blocks/warning/warning.css
git commit -m "Warning block: stronger border, dark text"
git push
    Block files bypass the transform Action entirely and go through aem.live's code-bus directly. Within 5–15 seconds the new version is served on aem.page.
  6. Hard-refresh the page in your browser to defeat the local cache (Cmd+Shift+R / Ctrl+Shift+R).The block now reflects your edits.
  7. (Optional) Add a new block.
    1. Create blocks//.css and blocks//.js.
    2. Update scripts/transform-dita.mjs to map a DITA pattern to a <div class="<new-block>"> in the cleaned HTML.
    3. Commit, push, and trigger a republish from AEM Guides so the transform runs against fresh DITA-OT output.
  8. (Optional) Update site-wide styling.Site-wide styles live in styles/styles.css. The boilerplate's base styles are at the top of the file; we keep a "POP modernization layer" at the end so it's easy to find what we changed.

    The brand tokens are defined as CSS custom properties at :root:

    :root {
  --pop-color-primary:  #1f3c88;
  --pop-color-accent:   #ff7849;
  --pop-color-warning:  #f59e0b;
  --pop-color-caution:  #ef4444;
  --pop-color-tip:      #10b981;
  --pop-color-text:     #1a1a1a;
  --pop-color-muted:    #6b7280;
  --pop-warning-tint:   #fff7ed;
  --pop-tip-tint:       #ecfdf5;
  --pop-caution-tint:   #fef2f2;
}

    Once these are swapped for the official POP palette, every block that references them gets the new look automatically.

You now know the loop for iterating on the EDS frontend: edit locally, push, hard-refresh. Block files propagate in seconds; topic content takes a republish from AEM Guides because it flows through the transform Action.
