Known issues and troubleshooting

The AEM Guides to EDS integration involves four independent systems, and each has at least one sharp edge. This topic catalogues the issues we hit while building the POC, each with its root cause and the workaround we use today.

You followed the setup tasks but somewhere along the pipeline content is not flowing as expected. The symptoms below describe the specific issues we have encountered, each paired with what causes the symptom and how to work around it.

The issues are ordered by where they sit in the pipeline, from AEM-side problems at the top to aem.live-side problems at the bottom.

Symptom: Generate Output fails with the error "Configure microservice on your cloud instance to use output type: FRANKLIN".

Cause: The publishing microservice OSGi configuration is not deployed to the AEMaaCS environment you are publishing from. The local AEM SDK does not include the microservice at all, and AEMaaCS environments without the OSGi configs cannot produce FRANKLIN output.

AEM developer

  1. Confirm the target environment is AEMaaCS, not a local SDK instance.
  2. Check that the two OSGi configuration files described in Deploy microservice OSGi configurations are committed to your AEM project and present in the deployed package.
  3. Verify the SERVICE_ACCOUNT_DETAILS Cloud Manager secret is set on the environment and contains valid Developer Console JSON.

Symptom: Clicking the publish profile Login button in AEM Guides opens a popup to GitHub, then hangs on Connection Failed after authorization. The popup closes but AEM never receives the auth result.

Cause: Adobe's Beta JavaScript opens a popup to github.com/login/oauth/authorize and waits for the popup's load event from the opener. Modern Chromium browsers enforce Cross-Origin-Opener-Policy (COOP) — when GitHub returns a COOP: same-origin header, the browser severs the opener-popup link and the load listener never fires. The same behavior is reproducible on Edge, Brave, and Arc.

AEM admin or developer

  1. Bypass the OAuth flow by creating the publish profile via the API, using the save-eds-publish-profile-aemaacs.sh script.The script POSTs to /bin/guides/v1/thirdparty/profile with a GitHub Personal Access Token in place of the OAuth result.
  2. Use the API path documented in Configure a publish profile and output preset.

Symptom: aem.page returns "failed to load /index.md from content-bus" even though the microservice committed HTML to your repository and the registration looks correct. Directly fetching from the worker URL returns HTTP 404 with body error code: 1042.

Cause: Adobe's hosted dita-franklin-worker Cloudflare Worker makes an internal fetch() to another worker on the adobeaem.workers.dev zone. Cloudflare blocks that call with error 1042 ("Worker tried to fetch from another Worker on the same zone, which is disallowed for security reasons"). The fix requires Adobe to enable the global_fetch_strictly_public compatibility flag or restructure the internal fetch.

EDS developer

  1. Bypass the worker by serving the microservice's HTML output through GitHub Pages and pointing fstab there.See Create and configure the EDS repository for the full procedure.
  2. If you need confirmation the issue is Adobe-side, capture the curl response:
    curl -i "https://dita-franklin-worker.adobeaem.workers.dev///main/docs/index.htm"

    A response containing error code: 1042 confirms the same-zone fetch restriction.

  3. (Optional) File an Adobe support ticket requesting the global_fetch_strictly_public compatibility flag on dita-franklin-worker.

Symptom: aem.page registration shows the wrong mountpoint URL, even though fstab.yaml in the repository contains the correct URL. Removing and reinstalling AEM Code Sync does not refresh the value.

Cause: AEM Code Sync snapshots fstab.yaml at install-time and stores the mountpoint URL in aem.live's persistent registration. There is no public API to refresh that snapshot — not commits, not webhooks, not removing and re-adding the repository, not even uninstalling and reinstalling the GitHub App.

Caution
Renaming the existing repository can in theory force aem.live to drop the registration, but in practice it introduces downstream registration weirdness. Starting from a fresh repository is the more reliable recovery.

EDS developer

  1. Create a fresh repository with the corrected fstab.yaml committed before installing AEM Code Sync.
  2. Update the AEM Guides publish profile to point at the new repository name.
  3. (Optional) Archive the old repository to avoid confusion.

Symptom: The pipeline completes, GitHub Pages serves the HTML, but aem.page renders empty pages (just the EDS chrome with no body content).

Cause:helix-html2md requires a very specific HTML shape: <main><div>…</div></main>. Two hard rules apply (verified by reading src/html2md.js in adobe/helix-html2md):

  • if (!select('main', hast)) return ''; — no <main> element means empty markdown.
  • createSections() iterates <main>'s direct children and removes any non-<div> siblings — so bare <h1> / <p> children of <main> get stripped.

EDS developer

  1. Confirm your transform script outputs the magic shape: <main><div>…all content…</div></main>.
  2. Run the transform output through @adobe/helix-html2md locally to verify it produces non-empty markdown:
    npm install --no-save @adobe/helix-html2md
node -e "import('@adobe/helix-html2md').then(({default: html2md}) => \
  html2md('
    Test
    ') \
  .then(r => console.log(r.md)))"
  3. Inspect the cleaned HTML in the repository under /docs/ to confirm the transform Action ran successfully.

Symptom: Pages with extension-less URLs (e.g., /contents/topics/overview) return 404 from GitHub Pages even though the file exists in /docs/.

Cause: GitHub Pages resolves extension-less URLs to .html files automatically but does not resolve them to .htm files. DITA-OT emits .htm by default.

EDS developer

  1. Ensure the transform Action renames every .htm file to .html after cleanup.The reference implementation in scripts/transform-dita.mjs does this with file.replace(/\.htm$/, '.html').

Symptom: Every DITA table renders as a differently-named block, with the first column header consumed as the block name.

Cause: aem.live treats every <table> as a block and uses the first cell text as the class name. A data table whose first column header is "Metric" becomes <div class="metric">, and the header row is consumed as the block name.

EDS developer

  1. Make sure the transform Action wraps every DITA-OT table as <div class="table"> with rows as nested divs, before helix-html2md sees them.
  2. Add a blocks/table/table.js that promotes the <div class="table"> structure into a real <table> with <thead> and <tbody> in the browser.See Customize EDS blocks and styling for the reference implementation.

Symptom: The sidenav appears for the most recent map you published, but pages from older maps in the same repository have no working navigation — clicking around shows the new map's nav even on old pages.

Cause: AEM Guides regenerates /blocks/sidenav/sidenav.js and sidenav_data.js on every publish, wholesale replacing the previous map's navigation. The boilerplate is fundamentally one-map-per-site.

Content engineer

  1. For multi-map publications, create a master ditamap that uses <mapref> to compose the child maps. Publish the master map instead of the child maps individually.
  2. (Optional) Use one EDS repository per publication if the maps are entirely unrelated.

Symptom: A page on aem.page still shows old content even after the transform Action has pushed cleaned HTML.

Cause: aem.live's content-bus caches the markdown representation of every page and only refetches when it sees a source-modified signal. In rare cases the signal is missed (e.g., rapid back-to-back publishes can race).

Anyone

  1. Force a preview refresh from the admin API:
    curl -X POST \
  -H "Content-Length: 0" \
  "https://admin.hlx.page/preview///main/"
