Create Adobe Developer Console credentials

The publishing microservice authenticates to Adobe IMS using an OAuth Server-to-Server credential. This task walks through creating that credential in Adobe Developer Console so AEM can call the microservice on your behalf.

Before you begin:

  • You are a System Administrator on the Adobe organization that owns the AEMaaCS environment, so you can create Developer Console projects and rotate credentials.

  • You have access to an AEM as a Cloud Service environment (author tier) where you have at least Developer permissions.

  • Your Adobe organization has been allowlisted by Adobe for the AEM Guides publishing microservice. (If not, contact your Adobe customer success manager.)

This is Track 1 of the three-track setup. The credential you create here is referenced by the OSGi config you deploy in Deploy microservice OSGi configurations, via a Cloud Manager secret.

The credential lives at the Adobe organization level, not the environment level. The same credential works for every AEMaaCS environment (dev, stage, prod) under that organization.

  1. Navigate to developer.adobe.com/console and sign in with the Adobe ID that has System Administrator on your target Adobe organization.If you have multiple Adobe organizations, switch to the correct one using the organization picker in the top right of the Console.
  2. Click Create new project and choose Project from template is not needed — pick the blank Empty project option.An empty project opens with a single, autogenerated workspace named Production.
  3. Rename the project to something descriptive, for example EDS Guides.This name only appears inside Developer Console, so it's safe to use whatever convention your team prefers.
  4. Inside the workspace, click Add API and select I/O Management API.
    1. On the credential type screen, choose OAuth Server-to-Server.Do not choose the deprecated JWT Service Account option, even if it's still shown in the UI.
    2. Give the credential a descriptive name such as EDS-Github-Credential and click Save configured API.
    The credential appears under the workspace's Credentials section with a Client ID, Client Secret, and other fields.
  5. Open the credential and copy the service account JSON from the Generate access token view.Look for the View service account JSON link. The JSON object contains every field the microservice needs: client_id, client_secret, the IMS endpoint, and the technical account fields.

    The JSON has this shape (do not commit the real values anywhere public):

    {
  "client_id": "abc123...",
  "client_secret": "p8e-...",
  "technical_account_id": "...",
  "technical_account_email": "...",
  "ims_org_id": "...",
  "scopes": ["AdobeID", "openid", "..."]
}
  6. Save the JSON to a password manager or other secure secret store. You will reference it from Cloud Manager in the next task.

You now have a working OAuth Server-to-Server credential that can authenticate to Adobe IMS and call the publishing microservice. The service account JSON is the single artifact that captures everything the microservice needs.

What an IMS Org ID looks like

The ims_org_id field has a recognizable shape — letters and digits, an @ sign, then AdobeOrg. Example (POP's IMS Org ID, which is not secret):

4A661E516570EABB0A495F88@AdobeOrg

Next, configure the AEMaaCS side so AEM can consume this credential at runtime: continue with Deploy microservice OSGi configurations.
