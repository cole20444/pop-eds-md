Create and configure the EDS repository

The EDS repository is where the publishing microservice commits generated HTML and where your frontend code (blocks, styles, scripts) lives. This task walks through creating the repository, configuring fstab.yaml, enabling GitHub Pages, and installing AEM Code Sync — in that order.

PREREQUISITE
Before you begin:
  • You have access to an AEM as a Cloud Service environment (author tier) where you have at least Developer permissions.
  • You have completed Deploy microservice OSGi configurations on at least one AEMaaCS environment.
  • You have git installed locally and can clone repositories you own.
The order of steps in this task matters. AEM Code Sync — the GitHub App that registers your repository with aem.live — takes a snapshot of fstab.yaml when it is installed and stores the mountpoint URL in aem.live's persistent registration. There is no public API to refresh that snapshot: not commits, not webhooks, not removing and re-adding the repository, not even uninstalling and reinstalling the GitHub App.
If you install Code Sync before fstab is correct, the only known recovery is to rename the repository (which has its own downstream complications). Save yourself the pain by following the order below.
  1. Create a new GitHub repository from the aem-guides-boilerplate template.
    1. Open the template repository and click Use this template.
    2. Choose a descriptive repository name. The convention is <product>-eds-<suffix> (for example pop-eds-md).
    3. Set the repository visibility to Public.
      ADDITIONAL INFORMATION: EDS requires the repository to be public so aem.live can fetch from it.
    4. Click Create repository from template.
  2. Clone the new repository locally.
    git clone https://github.com/<owner>/<repo>.git

    cd <repo>

  3. Edit fstab.yaml in the repository root so it points at the GitHub Pages URL your repository will serve.
    ADDITIONAL INFORMATION: The Pages URL follows the pattern https://<owner>.github.io/<repo>/. For POP's repository it is https://cole20444.github.io/pop-eds-md/.
    The final fstab.yaml should look like this (replace the URL with your own):
    mountpoints:

      /:

        url: https://cole20444.github.io/pop-eds-md/

        type: markup





    ADDITIONAL INFORMATION:
    Warning: Do not remove the mountpoint entirely. aem.live rejects an empty fstab with [admin] invalid fstab and will not register the site.
  4. Commit and push the corrected fstab.
    git add fstab.yaml

    git commit -m "Set fstab mountpoint to GitHub Pages URL"

    git push



  5. Enable GitHub Pages on the repository.
    1. In GitHub, navigate to SettingsPages.
    2. Under Source, select Deploy from a branch.
    3. Select the main branch and the /docs folder, then click Save.
    STEP RESULT: GitHub builds and deploys an initial empty Pages site within about 30 seconds. The URL appears at the top of the Pages settings panel.
  6. Verify Pages is serving by opening the Pages URL in a browser.
    ADDITIONAL INFORMATION: At this stage the page will be a 404 because /docs/ is empty, but the github.io hostname should resolve. A complete failure to resolve usually means Pages has not finished building yet.
  7. Install the AEM Code Sync GitHub App on the repository.
    ADDITIONAL INFORMATION: AEM Code Sync is the bridge between GitHub and aem.live. Installing it registers your repository with aem.live and provisions preview and live URLs.
    ADDITIONAL INFORMATION:
    Important: This step is the one that captures fstab.yaml. Make sure your commit from step 4 has landed on main before you click Install.
  8. Verify the registration by visiting the preview URL.
    ADDITIONAL INFORMATION: The preview URL has the form https://main--<repo>--<owner>.aem.page/. A successful registration shows a 404 or 200 page served by aem.page (not a hostname error). You can inspect the registration directly:
    curl -s "https://admin.hlx.page/status/<owner>/<repo>/main/" | jq
    The response should include a preview object whose sourceLocation matches your Pages URL, prefixed with markup:.
Result
Your repository is now wired into aem.live's registration. The next push to /docs/ will trigger an aem.live re-ingest, and the resulting markdown will be served at the preview URL.
At this point the repository contains only the boilerplate content. Real content arrives once the publishing microservice runs and the transform Action cleans up its output — that is the next task.

The four URLs you now have

URL
What it serves
https://github.com/<owner>/<repo>
The source of truth — code reviews, issues, actions.
https://<owner>.github.io/<repo>/
GitHub Pages, serving /docs/.
https://main--<repo>--<owner>.aem.page/
aem.page preview — what authors use to check unpublished changes.
https://main--<repo>--<owner>.aem.live/
aem.live production — what readers see.
