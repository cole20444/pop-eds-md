Publish a DITA map to EDS

With the credentials, OSGi configs, repository, transform Action, and publish profile all in place, publishing a map is a single click in AEM Guides. This task walks through that click, how to verify each stage of the resulting pipeline run, and how to force a refresh if something looks wrong.

PREREQUISITE
Before you begin:
  • All five setup tasks have been completed.
  • The ditamap you want to publish has an Edge Delivery Service output preset bound to a working publish profile.
  • You have access to an AEM as a Cloud Service environment (author tier) where you have at least Developer permissions.
Publishing is the moment the entire pipeline comes alive. The walkthrough below covers the happy path and points at verification steps you can run at each stage if you want to confirm the pipeline is healthy.
  1. Open the ditamap you want to publish in AEM Guides Map Console.
  2. Switch to the Generate Output tab.
  3. Select your EDS output preset and click Generate.
    ADDITIONAL INFORMATION: If the preset is greyed out, double-check that the publish profile is still selected on the preset's Config tab.
    STEP RESULT: A toast notification confirms the publish is queued. The microservice job runs asynchronously on Adobe's side.
  4. Wait for the success notification.
    ADDITIONAL INFORMATION: Depending on map size, expect 30–90 seconds. The UI shows the result inline, with a link to the aem.live URL it generated.
    STEP RESULT:
    You should see a message like main--<repo>--<owner>.hlx.live generated successfully.
    Note: The hlx.live hostname in the success message is a synonym for aem.live. Both forms work.
  5. (Optional) Verify the microservice committed to GitHub.
    ADDITIONAL INFORMATION: Open the repository's commits view. You should see a recent commit named Publishing for franklin.
  6. (Optional) Verify the transform Action ran.
    ADDITIONAL INFORMATION: In the GitHub repository's Actions tab, the most recent run of Transform DITA HTML for EDS should be green. The summary lists how many files were modified.
    STEP RESULT: A few seconds after the Action completes, a Clean DITA HTML for EDS rendering commit appears in the repository.
  7. Open the aem.page preview URL in a browser.
    ADDITIONAL INFORMATION: The URL has the form https://main--<repo>--<owner>.aem.page/<path>. For POP's pipeline the homepage is:
    https://main--pop-eds-md--cole20444.aem.page/
    STEP RESULT: The map's homepage renders with EDS chrome, the sidenav reflects the ditamap structure, and each linked topic loads from /contents/topics/....
  8. (Optional) Force a preview refresh from the admin API.
    ADDITIONAL INFORMATION: If aem.page is showing stale content, you can invalidate the content-bus cache directly:
    curl -X POST \

      -H "Content-Length: 0" \

      "https://admin.hlx.page/preview/<owner>/<repo>/main/<path>"



    ADDITIONAL INFORMATION:
    Tip: For a single page, append the page path (e.g., /contents/topics/foo). For the whole site, omit the path; aem.live recrawls everything.
  9. (Optional) Promote preview to live.
    ADDITIONAL INFORMATION: If your output preset had Push to live unchecked, you can still promote to aem.live after the fact through the admin API:
    curl -X POST \

      -H "Content-Length: 0" \

      "https://admin.hlx.page/live/<owner>/<repo>/main/<path>"



Result
The map is now live on aem.live and is being served from the global edge. End users see the content at https://main--<repo>--<owner>.aem.live/.
Subsequent edits to topics follow exactly the same workflow: edit in AEM Guides → click Generate Output → wait for the cascading commits → refresh the browser.

What a healthy admin status response looks like

The admin status API gives you a single view of the pipeline's state for a given path. The response is voluminous but the fields that matter for verification are:
{

  "preview": {

    "status": 200,

    "sourceLocation": "markup:https://cole20444.github.io/pop-eds-md/...",

    "lastModified": "..."

  },

  "live": {

    "status": 200,

    "sourceLocation": "...",

    "lastModified": "..."

  }

}





















Anything other than 200 on preview.status is a red flag — see Known issues and troubleshooting for the common causes.
