Architecture and pipeline

The AEM Guides to Edge Delivery Services integration is a five-stage pipeline that moves DITA content from authoring all the way to a browser. This topic describes each stage, the components involved, and the design decisions behind the shape we landed on.

The five-stage pipeline at a glance

Authoring happens in AEM Guides. Everything between Generate Output and the reader's browser is automated, and each stage has a clear contract with the next.

Stage 1 — Authoring (AEM Guides)

Content lives in AEM Guides as DITA topics referenced from one or more ditamaps. Authors work inside the Guides web UI or a connected XML editor.
The unit of publishing is the map: when you click Generate Output on a map, AEM bundles every topic it references, plus any conrefs and images, and hands the bundle to the publishing microservice.
The output profile chosen on the map's Output Presets tab tells Guides which kind of output to generate (HTML5, PDF, EDS, etc.) and where to send it. For this pipeline the relevant choice is Edge Delivery Service (Beta) bound to a publish profile pointing at your GitHub repository.

Stage 2 — Publishing (Adobe microservice)

The Adobe-hosted publishing microservice receives the bundle, runs DITA-OT inside an isolated container, and commits the resulting HTML files directly to your GitHub repository.
For the FRANKLIN output type the microservice writes:
  • /docs/index.htm — the map's homepage.
  • /docs/contents/topics/*.htm — one HTML file per topic referenced in the map.
  • /blocks/sidenav/sidenav.js and sidenav_data.js — the navigation tree extracted from the ditamap, baked into JavaScript constants.
  • /docs/nav.html and /docs/footer.html — site chrome boilerplate.
Caution: The microservice owns those file paths. If you hand-edit any of them, your edits will be silently overwritten on the next publish. Treat the /docs/ tree as machine-generated and make customizations elsewhere in the repository.

Stage 3 — Transform (GitHub Action)

DITA-OT's HTML is not in the shape that the helix-html2md converter expects. The .github/workflows/transform-dita.yml Action runs on every push to docs/**.htm and shells out to scripts/transform-dita.mjs, a cheerio-based cleanup script.
The transform performs these operations in order:
  1. Strip DITA-OT cruft such as prolog metadata, breadcrumbs, and prefix labels.
  2. Rewrite div-based tables as <div class="table"> EDS blocks.
  3. Normalize <code class="codeblock"> into <pre><code> with collapsed newlines.
  4. Map note variants (note_warning, note_tip, etc.) to clean <div class="warning|tip|caution|important"> blocks.
  5. Unwrap purely structural divs and spans so the resulting HTML has minimal nesting.
  6. Wrap the body in the <main><div>…</div></main> shape that helix-html2md requires.
  7. Rename .htm files to .html so GitHub Pages can resolve extension-less URLs.
The Action then commits the cleaned files back to the same /docs/ directory with the message Clean DITA HTML for EDS rendering.

Stage 4 — Hosting (GitHub Pages)

GitHub Pages is configured to serve /docs/ on the main branch. The resulting URL — https://<owner>.github.io/<repo>/ — is what fstab.yaml points at.
Why GitHub Pages instead of Adobe's hosted dita-franklin-worker? The worker currently returns Cloudflare error 1042 because of a same-zone fetch restriction it cannot satisfy on its own. See Known issues and troubleshooting for the full diagnosis.
GitHub Pages costs nothing for public repositories, builds in ~30 seconds after a push, and gives us complete control over the source aem.live ingests.

Stage 5 — Delivery (aem.live and EDS)

aem.live receives a preview or publish trigger, reads the mountpoint URL from fstab.yaml, and fetches HTML from GitHub Pages. The HTML flows through:
  1. helix-html2md — converts the HTML body into markdown.
  2. content-bus — caches the markdown keyed by the request path (the webPath).
  3. Markdown-to-HTML rendering — wraps the content in EDS chrome (<header>, <main> with <div class="section">, and <footer>).
  4. Client-side decoration — scripts/scripts.js runs in the browser, decorates each block, and injects the sidenav.

Why this particular shape

Three architectural decisions deserve a brief explanation. The full rationale lives in Known issues and troubleshooting.

Design decisions and reasoning

Decision
Reason
Use GitHub Pages instead of dita-franklin-worker
The worker returns Cloudflare error 1042 due to a same-zone fetch restriction. The fix is Adobe-side; GitHub Pages is a free, stable workaround we fully control.
Run a transform Action instead of feeding html2md raw DITA-OT HTML
helix-html2md requires a <main><div>…</div></main> body shape. DITA-OT emits bare <h1> and <p> children of <body>; without the transform html2md returns empty markdown.
Wrap every DITA table as <div class="table">
aem.live treats every <table> as a block named after the first-cell text. Wrapping every data table in a "table" div gives us one consistent block name across all tables.

End-to-end latency budget

From the moment an author clicks Generate Output to the moment a reader sees the change, expect roughly:

Typical duration
